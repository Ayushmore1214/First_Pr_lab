The DevOps Glossary

Plain-English definitions for the terms you keep hearing in cloud native conversations. No fluff, no jargon — just clear explanations for real beginners.

🔍
20 terms covered · Plain English definitions · CNCF ecosystem focused
Core Concepts

DevOps

A culture and set of practices that brings software development (Dev) and IT operations (Ops) teams together. Instead of developers throwing code over a wall for operations to deploy, everyone works together throughout the entire software lifecycle — from planning to production.

Culture Collaboration
Core Concepts

Cloud Native

An approach to building and running applications that takes full advantage of cloud computing. Cloud native apps are designed to be scalable, resilient, and manageable — typically built using containers, microservices, and automation from the ground up.

CNCF Scalability
Core Concepts

Microservices

An architectural style where an application is broken into small, independent services that each do one thing well. Instead of one giant app (a monolith), you have many small services that talk to each other over a network — making each one easier to deploy, scale, and understand.

Architecture Scalability
Core Concepts

Infrastructure as Code (IaC)

Managing and provisioning infrastructure (servers, networks, databases) using code and configuration files instead of manual processes. Tools like Terraform let you describe your entire infrastructure in a file, version it in Git, and apply it automatically — just like deploying software.

Terraform Automation
Core Concepts

GitOps

A way of managing infrastructure and deployments where Git is the single source of truth. When you push a change to Git, an automated system detects it and applies the change to your environment. This means every change is tracked, reversible, and peer-reviewed before it ever touches production.

Git Automation
Core Concepts

SRE

Site Reliability Engineering — Google's approach to running large-scale systems reliably. SREs apply software engineering principles to operations problems. They measure reliability with SLOs (Service Level Objectives), manage error budgets, and automate away toil (repetitive manual work).

Reliability SLOs
Containers

Container

A lightweight, standalone package that contains everything needed to run a piece of software — the code, runtime, libraries, and settings. Think of it like a shipping container: the same container runs identically on your laptop, a test server, or in production, no matter what the host environment looks like.

Docker Isolation
Containers

Docker

The most popular platform for building and running containers. Docker provides tools to package your application into a container image, share it via a registry like Docker Hub, and run it anywhere. A Dockerfile is the recipe that describes how to build your container image.

Dockerfile Container Runtime
Containers

Container Registry

A storage and distribution system for container images — like npm for JavaScript packages, but for containers. Docker Hub is the most well-known public registry. Teams also use private registries (like AWS ECR or GitHub Container Registry) to store proprietary images securely.

Docker Hub Images
Containers

Service Mesh

A dedicated infrastructure layer that handles communication between microservices. Instead of each service managing its own networking logic (retries, timeouts, encryption, load balancing), a service mesh like Istio or Linkerd handles it all transparently — like a smart network between your services.

Istio Linkerd
Kubernetes

Kubernetes

An open source system (originally from Google, now CNCF) for automating the deployment, scaling, and management of containerized applications. Often called K8s for short (there are 8 letters between K and s). Think of it as an operating system for your cluster of machines.

K8s Orchestration
Kubernetes

Pod

The smallest deployable unit in Kubernetes. A pod wraps one or more containers that need to work together and share resources. You rarely create pods directly — instead, you create higher-level objects like Deployments that manage pods for you, restarting them if they crash.

Kubernetes Workload
Kubernetes

Namespace

A way to divide a Kubernetes cluster into virtual sub-clusters. Teams or projects each get their own namespace, keeping their resources isolated. It's like having separate rooms in a shared office — same building, but your stuff doesn't mix with everyone else's.

Isolation Multi-tenancy
Kubernetes

Helm

The package manager for Kubernetes — like apt for Ubuntu or npm for Node.js. Helm packages Kubernetes manifests into reusable "charts" that can be versioned, shared, and deployed with a single command. It handles complex applications that need many Kubernetes resources working together.

Charts Package Manager
Kubernetes

Ingress

A Kubernetes resource that manages external access to services inside a cluster — usually HTTP and HTTPS traffic. An Ingress controller (like nginx or Traefik) reads Ingress rules and routes traffic to the right service. Think of it as the cluster's front door and traffic director.

Networking Routing
CI/CD

CI/CD

Continuous Integration and Continuous Delivery (or Deployment). CI means automatically building and testing code every time someone pushes a change. CD means automatically deploying that tested code to production (or a staging environment) — so software releases happen frequently and safely, not as rare, stressful events.

Automation Pipeline
CI/CD

Pipeline

An automated sequence of steps that code travels through from commit to production. A typical pipeline might: build the code, run tests, check for security vulnerabilities, build a container image, push it to a registry, and deploy it. Each step must pass before the next one begins.

Automation CI/CD
Observability

Observability

The ability to understand what's happening inside your systems from the outside — by looking at the data they produce. Observability has three pillars: Logs (timestamped records of events), Metrics (numeric measurements over time), and Traces (the path a request takes through your system).

Logs Metrics Traces
Observability

Chaos Engineering

The practice of intentionally breaking things in production to find weaknesses before they cause real outages. Netflix pioneered this with "Chaos Monkey" — a tool that randomly kills servers in production. The idea: if you regularly test failure, your system gets better at handling it.

Resilience Netflix
Observability

Serverless

A cloud execution model where you run code without managing servers. You write a function, deploy it, and the cloud provider handles provisioning, scaling, and billing — you only pay for the exact compute used. AWS Lambda, Google Cloud Functions, and Azure Functions are popular examples.

FaaS Cloud

How Contributing Works

  1. 1
    Intro PR Add your name to CONTRIBUTORS.md. Title: "My First Open Source Contribution"
  2. 2
    Pick a Real Issue Once your intro PR is merged, grab a good first issue from GitHub Issues.
  3. 3
    Open a PR Make your change, push it, and open a pull request. We'll review it together.
  4. 4
    Get Merged Your name goes into the project history forever. That's your first open source contribution.

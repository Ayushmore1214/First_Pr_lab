Open Source Glossary

Plain-English definitions for the terms you keep seeing before your first contribution. No assumed knowledge.

Git

Repository

The folder where a project lives - including every file and the full history of every change ever made. When you clone one, you download a complete copy to your machine. Often called a "repo."

  • repo
  • git clone
  • version control
Git

Fork

Your personal copy of someone else's repository, created on GitHub. You experiment freely in your fork - nothing in the original changes until you open a pull request. Think of it as making a photocopy before you start marking it up.

  • copy
  • upstream
  • contribute
Git

Branch

An isolated workspace inside a repository. You make changes on a branch so the main codebase stays untouched until your work is reviewed and ready. Every contribution should live on its own branch.

  • git checkout -b
  • main
  • isolation
Git

Commit

A saved snapshot of your changes, stored permanently in the project history. Each commit has a short message describing what changed. Good commit messages explain the why, not just the what.

  • git commit
  • history
  • sign-off
Git

Merge

Combining the changes from one branch into another. When a maintainer approves your pull request, they merge your branch into main. Your changes become a permanent part of the project.

  • git merge
  • main
  • pull request
Git

Clone

Downloading a repository from GitHub to your local machine. You clone your fork, not the original repo - that way you have write access. Run git clone <url> to get the files and the full history.

  • git clone
  • local
  • download
GitHub

Pull Request

A formal proposal to add your changes to a project. You open a PR, a maintainer reviews it, leaves feedback or approves it, and merges it. This is the core workflow of open source contribution. Often called a PR.

  • PR
  • review
  • merge
GitHub

Issue

A tracked conversation on GitHub - a bug report, feature request, question, or task. Issues are how a project organizes its work. You should always look for an issue to claim before starting work on a change.

  • bug report
  • feature request
  • good first issue
GitHub

Code Review

The process of reading someone else's changes and leaving feedback before they're merged. It's collaborative, not critical. Reviewers might ask questions, suggest improvements, or approve the changes as-is.

  • feedback
  • approve
  • changes requested
GitHub

Upstream

The original repository that you forked from. When the original gets new commits, you pull from upstream to bring those changes into your fork and stay in sync. You set this up with git remote add upstream <url>.

  • git remote
  • sync
  • origin
Project

Maintainer

The person (or team) responsible for a project. They review pull requests, merge changes, close stale issues, and set the direction. When you contribute, you're working with them, not for them.

  • owner
  • reviewer
  • merge access
Project

Contributor

Anyone who has added something to a project - code, documentation, a bug report, a typo fix, a translation. You become a contributor the moment your first change is accepted. This project is designed for that exact moment.

  • first PR
  • CONTRIBUTORS.md
  • open source
Project

README

The front page of a repository. It's the first thing most people read. A good README explains what the project is, how to run it locally, and how to contribute. Always read it before opening an issue or pull request.

  • README.md
  • docs
  • markdown
Project

License

The legal terms for what others can do with the code. MIT means do whatever you want, just keep the copyright notice. GPL means any project using the code must also be open source. Always check the license before using code in a project.

  • MIT
  • GPL
  • Apache
Process

DCO

Developer Certificate of Origin. A sign-off line at the end of every commit that says you wrote the code and have the right to contribute it. Add it with git commit -s - git appends Signed-off-by: Name <email> automatically.

  • sign-off
  • git commit -s
  • legal
Process

CI/CD

Continuous Integration and Delivery. Automated checks that run every time someone pushes code - building it, running tests, checking formatting. If a check fails you'll see a red X on your PR. This project uses CI to validate HTML and enforce DCO sign-offs.

  • GitHub Actions
  • pipeline
  • automated checks
Process

Linting

Automated checking of code for style or formatting mistakes - missing tags, inconsistent indentation, deprecated patterns. A linter runs in CI so issues get caught before a human ever has to point them out in review.

  • HTMLHint
  • code style
  • CI
Process

Open Source

Software whose source code is publicly available for anyone to read, use, modify, and contribute to. Most of the tools the world runs on - Linux, Git, Kubernetes, VS Code - are open source. This project was built so you can experience contributing to one.

  • public
  • community
  • contribute
